【システム設計仕様書】ステートレスPython CGIゲームの堅牢化リファクタリング
1. 基本前提（動作環境の制約）

本システムは、WSGI/FastAPI/Flaskなどの常駐Webアプリケーションサーバーを使用せず、サーバー制限により**「1リクエスト1プロセス」で動作する伝統的なCGI（Common Gateway Interface）環境**で動作します。

この制約（プロセス間でのメモリ共有不可、リクエストごとの起動オーバーヘッド、多重アクセス時の競合リスク）を克服しつつ、モダンなWeb標準とセキュアな堅牢性を両立させるため、以下の設計思想・アーキテクチャを採用します。
2. 4つのコア設計思想（Design Philosophy）
2.1. データの一元化とアトミックな永続化 (Atomic I/O & Single Data Source)

    個別ファイルI/Oの統合: ユーザーごとに「ステータス」「パーティ」「倉庫」「図鑑」などのファイルを個別に読み書きせず、1つの辞書（dict）オブジェクトに一元化し、単一のファイル（例: user_all.pickle）として1回のI/Oで処理を完結させます。これにより、ファイル読み書きのオーバーヘッドを削減し、不整合を防ぎます。

    アトミックな保存 (Atomic Write): 保存処理時は直接ファイルを上書きせず、同じディレクトリ内に一時ファイルを生成してデータを書き込み、同期（fsync）が完了した後に os.replace で元のファイルと置き換えます。これにより、書き込み中のプロセス強制終了によるファイル破損（0バイト化など）を完全に防止します。

2.2. 遅延ロードを伴う集中型ルーティング (Dynamic Routing & Lazy Loading)

    フロントコントローラーパターン: 全てのPOSTアクションは単一のエントリポイント（例: login.py）で一括して受け取ります。

    遅延インポート (Lazy Load): ルーターは、セッション・CSRF・入力バリデーションなどの共通前処理を通過した後、必要な機能モジュール（例: kyoukai.py などの個別ハンドラ）のみを importlib.import_module で動的に遅延ロードし、関数を実行します。起動時に使わない全モジュールをインポートするコストを排除し、CGIの応答速度を向上させます。

2.3. ディレクトリベースの排他ロック (Directory-based Locking)

    ステートレス環境でのMutex: exLock（ディレクトリの os.mkdir の原子性を利用した排他制御）を採用します。

    2階層のロック管理:

        ユーザー個別ロック: 同一ユーザーの多重リクエスト（連打など）によるセーブデータ破損を防ぐためのロック。

        共有ファイルロック: user_list や掲示板ログなど、複数ユーザーが同時に読み書きする共有ファイルを守るための共有ロック。

2.4. 防衛的な入力値の標準化とバリデーション (Defensive Validation)

    WTFormsによる厳格なバリデーション: 文字種、長さ、型を検証します。

    ファイルシステム保護: ユーザー名がそのままフォルダ名やファイル名になる仕様の場合、 Windows/Linux などの予約語（CON, PRN, NUL など）や、ディレクトリトラバーサルを引き起こす制御文字、絵文字をバリデータ層で厳格に弾きます。

    Shift-JIS変換保証: レガシーな環境や古いブラウザとの互換性を保つため、内部処理はすべて「UTF-8」で行いつつ、ユーザー入力が Shift-JIS に変換可能であることをバリデーション段階で検証し、文字化けやファイル名作成エラーを防ぎます。

4.1. 動的ルーティング (login.py 相当)
code Python

import cgi, importlib, sys
from sub_def.crypto import get_session, token_check
from sub_def.utils import error

FUNCTION_MAP = {
    "my_page": ("cgi_py.my_page", "my_page_view"),
    "shop_buy": ("cgi_py.shop", "buy_item"),
}

def main():
    form = cgi.FieldStorage()
    FORM = {k: form.getfirst(k) for k in form}

    # 共通セッション・CSRF検証
    session = get_session()
    FORM["s"] = token_check(FORM, session)

    mode = FORM.get("mode")
    if mode not in FUNCTION_MAP:
        error("無効なモードです")

    module_path, func_name = FUNCTION_MAP[mode]
    # 必要なモジュールのみを遅延インポートして実行
    module = importlib.import_module(module_path)
    func = getattr(module, func_name)
    func(FORM)

if __name__ == "__main__":
    sys.stdout.reconfigure(encoding="utf-8")
    main()

4.2. 排他ロックとアトミック保存 (file_ops.py 相当)

import os, tempfile, pickle, exLock
from typing import Any

def save_data_atomically(data: Any, file_path: str, lock_name: str) -> None:
    # 独自ディレクトリロック
    lock = exLock.exLock(os.path.join("./locks", lock_name))
    if not lock.lock():
        raise RuntimeError("Lock acquisition failed")

    dir_path = os.path.dirname(file_path)
    os.makedirs(dir_path, exist_ok=True)

    temp_file_path = None
    try:
        # 1. 同一フォルダ内に一時ファイルを生成
        with tempfile.NamedTemporaryFile(
            mode="wb", dir=dir_path, prefix=".tmp_", suffix=".pickle", delete=False
        ) as temp_file:
            # 2. 一時ファイルへの書き込み
            pickle.dump(data, temp_file, protocol=pickle.HIGHEST_PROTOCOL)
            temp_file.flush()
            os.fsync(temp_file.fileno())
            temp_file_path = temp_file.name

        # 3. アトミックな置換
        os.replace(temp_file_path, file_path)
    except Exception as e:
        if temp_file_path and os.path.exists(temp_file_path):
            os.remove(temp_file_path)
        raise e
    finally:
        lock.unlock()

    【他CGIゲームリファクタリング時のAI宛て指示文】

    あなたはプロのPythonバックエンドエンジニアです。
    提供するレガシーなCGIプログラムを、以下の**【ステートレスPython CGI堅牢化設計】**に従って Python 3 でリファクタリング、あるいは新規書き換えを行ってください。
    必須要件:

        データ一元化: ユーザー個別の複数ファイルを user_all.pickle という単一の辞書データに統合し、1回のファイルI/Oで全状態が同期されるように設計してください。

        アトミック保存: データの書き込みは、同一ディレクトリ内の一時ファイルへの出力後、 os.replace によるアトミック置換を行ってください（直接上書きによる0バイト化の完全な排除）。

        遅延インポートルーター: メインエントリポイントでリクエスト（POST）を受け取り、共通セッション/CSRFチェックを行った後、要求された機能だけを importlib で遅延インポートして実行させてください。

        排他制御の徹底: ユーザーごとの連打（多重リクエスト）競合や、共有データ（ユーザー一覧、グローバル変数、掲示板など）の同時書き込み競合を防ぐため、ディレクトリベースの排他ロックを関数呼び出しの最外殻で取得・解放してください。

        堅牢なバリデーション: WTForms等を用い、フロントエンドから送信された値を信頼せず型・長さ・許容値をチェックしてください。特に、ファイル名・フォルダ名に利用される値（ユーザー名など）については、Windows/Linux予約語や制御文字、Shift-JIS変換不可能な文字、絵文字を必ずバリデータで弾いてください。

        NoReturn関数の設計: 成功/エラー遷移を処理する関数（error, success, print_html）は、最終的に sys.exit() を呼んでプロセスを終了する NoReturn（制御を戻さない）関数とし、呼び出し側で return error() のようなデッドコードを記述させないでください。

    上記の設計原則に従い、保守性と堅牢性に優れたコードを、要所に日本語コメントを付与しながら出力してください。
