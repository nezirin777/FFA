# FFA改 Vips — ルーティング / コマンドフロー 保守ドキュメント

各ページ・コマンドが `login.py` の中央ディスパッチャからどのモジュール・テンプレートへ流れるかを追うための開発者向けマップ。ソースコード（`login.py` の `FUNCTION_MAP` と各モジュール）から抽出した実データに基づく。

> mermaid 図は GitHub / VS Code(Markdown Preview Mermaid拡張) 等で描画されます。

---

## ① 全体構成

エントリポイントは4つ。実処理はすべて `login.py` に集約され、`FUNCTION_MAP` が `cgi_py/*.py` の各モジュールへ振り分ける。データは `save_data/*.json` にファイルロック付きで読み書きする（DBなし）。

```mermaid
flowchart TD
  idx["index.html / frame1.html"] --> others["others.py 〈ログイン前トップ〉"]
  others -->|新規登録| cmake["chara_make.py 〈キャラ作成〉"]
  others -->|ログイン POST| login
  cmake -->|作成後 redirect| login
  login["login.py 〈中央ディスパッチャ〉"]
  login -->|mode=log_in POST| authn["CSRF照合 → パスワード検証 → Cookieセッション発行 → redirect mode=main"]
  login -->|?mode=X URLクエリ| FM{"FUNCTION_MAP"}
  FM --> mods["cgi_py/*.py 〈各機能モジュール〉"]
  mods --> tpl["templates/*.html 〈Jinja2 / autoescape〉"]
  mods <--> data[("save_data/*.json + exLock 排他ロック")]
  admin["admin.py 〈管理パネル・独立〉"] <--> data
  tpl --> view["ブラウザ表示"]
```

## ② リクエスト処理のライフサイクル

1リクエストが `login.py main()` に入ってから描画されるまでの共通パイプライン。認証・CSRF・認可（IDOR対策）のゲートを通過する。

```mermaid
flowchart TD
  req["リクエスト → login.py main()"] --> mres["mode 解決: URLクエリ ?mode= を最優先／無ければ POSTボディの mode"]
  mres --> islog{"mode == log_in ?"}
  islog -->|Yes / POST| lp["CSRF token_check → verify_password（新旧/平文対応）→ 旧形式なら再ハッシュして移行 → Cookie発行 → redirect main"]
  islog -->|No| pub{"公開mode? rank / system / chara_sts / img_list / ranking"}
  pub -->|未ログイン かつ 非公開| rdr["others.py へ redirect"]
  pub -->|通過| ispost{"POST ?"}
  ispost -->|Yes| csrf["CSRF token_check（FORM s ⇔ session csrf_token）"]
  ispost -->|No| disp
  csrf --> disp["FUNCTION_MAP[mode] を import → module.main()"]
  disp --> owner["require_owner(id)〈状態変更系のみ〉cookie本人 + パスワード一致 を要求"]
  owner --> sub["body mode で サブ動作を分岐"]
  sub --> out["render_template(...) / redirect(...)"]
```

**ゲートの意味**
1. 未ログインで非公開modeなら `others.py` へリダイレクト。
2. POST は必ず **CSRF** 照合（ワンタイムトークン `s`）。
3. 状態変更系は `require_owner` で「他人の `?id=` による操作（IDOR）」を遮断。

## ③ 2層の `mode` 解決（このシステムの肝）

- **URLクエリの `mode` でモジュールを決定**
- **POSTボディの `mode` でモジュール内のサブ動作を分岐**
- `common.decode_params()` は GET+POST をマージし **POST 優先**

```mermaid
flowchart LR
  q["URLクエリ login.py?mode=bank"] -->|モジュール振り分け| M["cgi_py.bank に到達"]
  b["POSTボディ mode=bank_sell"] -->|サブ動作分岐| M
  M --> S["預入 / 引出 を実行"]
```

例: フォームは `action="login.py?mode=bank"`（振り分け）＋ hidden `<input name="mode" value="bank_sell">`（サブ動作）。config の `*_script` エイリアスはこの「`?mode=` 付き」を前提にしている。

## ⑤ 代表的なコマンドフロー

### 対人戦（王者挑戦） `mode=battle`
```mermaid
flowchart LR
  A["ffadventure.html 王者に挑戦(+s,id)"] -->|POST mode=battle| B["battle.py"]
  B --> C["require_owner + クールダウン確認"]
  C --> D["BattleSimulator 実行〈skills.py 必殺技〉"]
  D --> E["勝敗で gold/exp 更新・王者データ交代"]
  E --> F["monster_result.html"]
```

### ショップ購入・売却 `mode=shop_item`
```mermaid
flowchart LR
  A["shop_trade.html"] -->|POST mode=buy/sell| B["shop_item.py"]
  B --> C["require_owner"]
  C --> D{"buy?"}
  D -->|buy| E["購入 → 倉庫へ送付"]
  D -->|sell| F["装備を売却"]
  E --> G["shop_result.html"]
  F --> G
```

### 掲示板 投稿（PRGパターン） `mode=bbs`
```mermaid
flowchart LR
  A["bbs.html 投稿フォーム(+s)"] -->|POST mode=post| B["bbs.py"]
  B --> C["CSRF / require_owner / 禁止ワード / 文字数"]
  C --> D["bbs.json に追記〈bbs_post ロック〉"]
  D -->|redirect| E["GET mode=bbs 一覧を再表示"]
```
